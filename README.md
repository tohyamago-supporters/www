# 遠山郷応援会 ウェブサイト (tohyamago.org)

長野県南信州の秘境「下栗の里」をはじめとする遠山郷の自然・文化・暮らしを守り、後世に豊かな社会を残すために活動する **一般社団法人遠山郷応援会** の公式ウェブサイトです。

> 本 README はこのプロジェクトの **人間向けドキュメント** です。サイトの **目的・ビジョン・情報設計・デザインシステム・コピー方針・ロードマップ** をまとめています。デプロイ / CI・ディレクトリ構造・データスキーマ・記事の追加手順といった **実装の細部（AI セッションを動かすための前提・ルール）** は [`CLAUDE.md`](./CLAUDE.md) にあります。

## 1. 私たちが目指すもの (Mission / Vision)

### ミッション

> **「秘境の息遣いと、守る人の想いを伝える」**

人口減少や資本流出によって衰退しつつある遠山郷において、

- 地域経済の基盤となる **農地・山林の維持保全**
- 地域内にある価値の **可視化・事業化**
- 地域の **魅力と課題の情報発信**

を行い、**自律継続的に人口の維持と経済活動の発展が行われる仕組み** をつくることを目的としています（→ [活動趣旨](./src/pages/purpose.astro)）。

地域住民だけでなく、都市部に住む人々が地域の課題を「自らに係るもの」として主体的に取り組める ―― そんな **地域内外で無理なく協力し合える環境** をウェブサイトを通じて広げていきます。

### サイトの役割

このサイトは、単なる団体紹介ページではなく、次の3つの機能を担います。

1. **記録 (Archive)** — 2018年から続く下栗芋・大豆・蕎麦・茶の作付けや地域交流の歩みを、写真と短い言葉で残し続ける（→ [`src/content/posts/`](./src/content/posts/)）。
2. **接点 (Connect)** — ボランティア・入会・成果品購入・寄付といった「関わりしろ」への入口を、迷わず提供する。
3. **支援 (Support)** — 都市部の余剰資本を地域へ還元する手段（会費・寄付・物販）を、安心して使える形で提供する。

### これまでの歩み（大切にしている流れ）

本サイトの記事は、Facebook 風の素朴な近況投稿として 2018 年から積み重ねられてきました。

- **2018〜2019**: 下栗芋・大豆の収穫、種まきなど季節の便り。短く飾らない言葉が中心。
- **2020〜2022**: 定点観測機器 (AgriRecorder) によるタイムラプス、ボランティアとの「ゆるいつながり」づくり、コロナ禍での試行錯誤。
- **2023〜2024**: 竹パウダーによる土壌改良など栽培の工夫、お茶の手摘み（一芯三葉）への挑戦と数量限定販売。
- **2025〜2026**: 新規参加者を交えた蕎麦の収穫・脱穀、茶畑の整枝、地域行事への参加と交流。

> リデザインにあたっても、この **「飾らず、続いてきた記録」** の温度感は失わないことを最優先とします。新しいデザインは、過去の歩みをより読みやすく・たどりやすくするためのものです。

## 2. ターゲットとゴール

サイト構成は **「誰に来てほしいか」「何をしてほしいか」** から逆算します。本サイトは主に **これから関わる新規来訪者** に向けた入口です（既に参加・寄付してくれている人への深い情報発信は、将来のマイページ — ロードマップ Phase 5 — に委ねます）。

### 想定ターゲット

| ターゲット                   | 像                                      | このサイトでの主目的                  |
| ---------------------------- | --------------------------------------- | ------------------------------------- |
| **ボランティア参加見込み**   | 比較的若い層（20〜40 代）。これから参加 | 農作業・イベントへの初参加 → 継続参加 |
| **支援見込み（購入・寄付）** | 富裕層・高齢層                          | 賛同しての成果品購入、寄付・入会      |

### ゴール優先順位

1. **継続的な（2 回以上）活動参加者を増やす ＝ コミュニティの拡大**（最優先）
2. 活動に賛同する気持ちでの **ショップ購入**
3. **継続寄付する会員**数の増加

> まずは **金額よりコミュニティの大きさ** を目指します。サイトの全導線は、この優先順位（参加 ＞ 購入 ＞ 寄付）の順に目立たせます。

### 「関わりしろ」の階段

来訪者を一足飛びに会員化しようとせず、軽い一歩から段階的に深い関与へ導きます。

```
知る（趣旨・遠山郷との始まり・活動記録）
  → 参加する（はじめての方へ・農作業カレンダー・交通案内 → activo 応募）   ← 最優先
    → 賛同する（成果品を買う）
      → 支える（継続寄付・入会）
```

## 3. サイト構成（情報設計とユーザー導線）

### 設計原則

- **来訪者の目的で章立てする**（団体組織図ではなく「やりたいこと」で分ける）。
- 各ページに **次の一歩（CTA）** を必ず置き、ゴール優先順位の高い「参加する／寄付」へいつでも戻れるようにする。
- 現状のコンテンツ（趣旨・入会案内・定款・公告・特商法・近況記事）は **すべて残す**。掲載場所・導線は再編する。

### 農作業カレンダーについて

本サイト独自の目玉機能。**二つの役割**を一つの画面で満たします。

1. **参加意欲の喚起** — 「いつ・どの作物の・どんな作業に参加できるか」を一望でき、各作業から参加導線（`/join`・activo）へ送る。
2. **運営の確認用台帳** — 作付け・作業・地域行事の年間スケジュールを、管理者が 1 ファイルで更新できるデータとして保持。

作物の作業（毎年循環・旬粒度）と地域イベント（特定日）をデータ化し、横軸＝月・縦軸＝作物のガントチャートで描画、地域イベントを重ねて表示します。各作業には**作業強度（軽め／ふつう／しっかり）**を持たせ、初参加者が負荷感を掴めるようにしています。詳細なデータ構造は [`CLAUDE.md`](./CLAUDE.md) を参照。

## 4. デザインシステム

デザインの方向性は **「秘境の息遣いと、守る人の想いを伝える」**。広大な自然の空気感（余白）と、山里の陽光・土・水の温かみを、緑（山）を基調に陽光イエロー・コーラル・空のティールで差した多色で表現します。

### 4.1 カラーパレット

Tailwind のテーマトークン（`global.css` の `@theme`）として定義し、`bg-base` のようなユーティリティで扱えます。**濃い派生（`-deep` / `-strong`）は白文字 CTA など WCAG AA を要する箇所用**、淡い派生（`-soft`）は帯・タグ・装飾用です。ハードコードした hex ではなく必ずこのトークンで指定します。

| トークン        | 値        | 用途                                            |
| --------------- | --------- | ----------------------------------------------- |
| `base`          | `#fffdf7` | あたたかいクリーム白。サイト全体の背景          |
| `surface`       | `#ffffff` | カード・コンテンツエリアの背景                  |
| `primary`       | `#1f855f` | 明るい深緑〜ティール。見出し・重要テキスト      |
| `primary-deep`  | `#1a7152` | 白文字 CTA / メニュー / 小さい強調文字（AA 用） |
| `primary-soft`  | `#44b08c` | primary の明るい派生。装飾・ボーダー            |
| `body`          | `#3e3a35` | 温かみのあるダーク。一般本文                    |
| `accent`        | `#f06b4b` | コーラル（朱）。装飾・アイコン・ボーダー        |
| `accent-strong` | `#cc4329` | accent の濃い派生。白文字 CTA（AA ≒ 4.6:1）     |
| `accent-soft`   | `#f7a892` | accent の淡い派生。カラーブロック・ホバー       |
| `sunlight`      | `#ffc24b` | 陽光イエロー。差し色・アクセントライン          |
| `sunlight-soft` | `#ffe6ac` | 陽光の淡い派生。帯・タグ                        |
| `sky`           | `#4eb1c4` | 空・水のティール。装飾                          |
| `sky-soft`      | `#d6eef1` | 空の淡い派生。グラデーション帯                  |

### 4.2 タイポグラフィ

| 役割                   | フォント                           | 意図                       |
| ---------------------- | ---------------------------------- | -------------------------- |
| 見出し・キャッチコピー | `font-serif`（Noto Serif JP 想定） | 情緒的・伝統的・誠実       |
| 本文                   | `font-sans`（標準ゴシック）        | 読みやすさ重視             |
| 数字・英字アクセント   | `font-mono`                        | モダンで活動的なニュアンス |

ベースのフォントウェイトは light（現行 `:root { font-weight: light }` を踏襲）。

### 4.3 レイアウトとスペーシング

- **コンテナ幅**: 最大 `max-w-6xl` / `max-w-5xl`、`mx-auto` で中央揃え。左右に必ず `px-4`〜`px-6`。
- **余白（空気感）**: セクション間は `py-16`〜`py-24` を推奨。秘境の広大さを表現するため、要素を詰め込みすぎない。
- **レスポンシブ**: モバイルファースト。PC 向けは `md:` / `lg:` プレフィックス。

### 4.4 UI コンポーネント規則

- **CTA ボタン**: 丸み（`rounded-full` / `rounded-xl`）を基本に。
  ```html
  <a
    href="/donate"
    class="bg-accent text-white px-8 py-3 rounded-full font-bold transition-all hover:opacity-90 hover:shadow-lg"
    >寄付する</a
  >
  ```
- **カード（記事・商品）**: わずかなシャドウ＋ホバーで浮き上がる。
  ```html
  <article
    class="bg-surface rounded-2xl shadow-sm border border-gray-100 overflow-hidden transition-transform hover:-translate-y-1 hover:shadow-md"
  >
    …
  </article>
  ```

### 4.5 コーディング必須要件

- セマンティックタグ（`<header>` / `<main>` / `<section>` / `<article>` / `<footer>`）を正しく使う。
- 画像には必ず適切な `alt` を設定する（※現行の記事画像は `alt=""` の装飾扱いが多いため、リデザイン時に意味のある代替テキストを検討）。
- コードはシンプルに保ち、不必要に深いネストを避ける。
- インタラクティブ部品は Astro Islands（React 19）。静的に描けるものは Astro コンポーネントで。

### 4.6 装飾モチーフ

- **山（▲）はブランドマークの核に限定**する。ロゴ（`SiteLogo.astro`＝遠山郷の重なる稜線＋陽光＋コーラルの頂）にのみ使い、サイト全体を山モチーフで埋め尽くさない。
- **見出し（`SectionHeading.astro`）は「三つの実」マーク**を使う。深緑 → 陽光 → コーラルの、上昇する 3 つの凹面ひし形で、実り・広がり・育つコミュニティを表す。色は必ずデザイントークン（`fill-primary` / `fill-sunlight` / `fill-accent`）で指定する。
- **バッジ（ラベルピル）は三角なしのベタピル**に統一する。基準は `rounded-full bg-primary-deep px-4 py-1 font-serif text-sm tracking-wide text-white`。暗い写真の上では `ring-1 ring-white/25` と `text-shadow` を足す。カードのタイトル行に添える小ステータス（例: 「なかま募集中」）はサイズと地色だけ縮める（`px-2.5 py-0.5 text-[0.65rem] font-medium`）。**三角なし・ベタ塗り・白文字・`rounded-full`** が核。

### 4.7 画像（モバイル回線の体感速度）

- `astro:assets` の `<Image>` を使い、`astro.config.mjs` の `image.layout='constrained'` ＋ `responsiveStyles` でレスポンシブ画像（`srcset` / `sizes`、出力は webp）を全画像で自動生成する。表示幅が小さい画像には `sizes` を明示する。
- ファーストビュー（LCP）の画像のみ `loading="eager"` ＋ `fetchpriority="high"`。それ以外は既定の `loading="lazy"`。
- 写真は **ブラー placeholder（LQIP / blur-up）** で表示し、体感の遅延感を消す。`BlurImage.astro`（`<Image>` のラッパー）が、ビルド時に生成した極小のぼかし下絵を即時表示し、本画像をその上へフェードインで覆う。低速回線でも白い空白やカクつきが起きない。

### 4.8 アクセシビリティ・パフォーマンス指針

- コントラスト比は WCAG AA を満たす（特に濃色トークン上の白文字、本文グレー）。色だけに依存せずラベル・凡例・`aria` を併用する。
- キーボード操作・フォーカスリングを確保（ナビ、決済導線、フォーム）。

### 4.9 コピー（文章）の方針

見出し・CTA・カード・リード文・ナビ項目など、運営が書くサイトの「地の文」全般に適用するトーン指針。

- **「行動」にフォーカスし、「畑」という場所・モノを強調しすぎない。** 私たちの本質は **景観・暮らし・文化を守り継ぐ活動と、そこに集う人のつながり** にある。「畑に参加する」→「活動に参加する」、「畑の一年」→「活動の一年」、「今、畑でできること」→「今月の活動」。「畑」「収穫」「お茶摘み」などの具体語は、活動内容を説明する文脈では引き続き使ってよい（前面の見出し・CTA で過度に繰り返さないのが趣旨）。
- 来訪者の **行動（参加する・知る・支える・買う）** を主語に据え、各ページに「次の一歩」の CTA を置く。
- **「お気軽に」は使わない。** 具体的な意思のない問い合わせ・参加を増やしかねないため避ける。遠山郷という秘境に辿り着けるのは芯のある人だけ、という活動の性質に合わせ端的に書く（「お気軽にお問い合わせください」→「お問い合わせください」）。
- **例外**: 既存の記事（`src/content/posts/`）の本文、および読み物として書かれた「遠山郷との始まり」（`/story`）の地の文は書き手の表現を尊重し、機械的な置換は行わない。画像の `alt` も事実の描写を優先する。

## 7. 移行方針（現行 → 新デザイン）

リデザインは **一括置換ではなく段階移行** で進めてきました。デザイントークン先行・共通プリミティブ整備・ページ単位の適用（1〜3）は Phase 1〜2 で完了済みで、残るは動的化（4）と記録の後方互換維持（5、恒久ルール）です。

1. **トークン先行**: まず Tailwind テーマに新カラー／フォントを追加し、新旧クラスを共存させる。
2. **プリミティブ整備**: Button / Card / Container などの共通部品を新システムで作り、ページごとに差し替える。
3. **ページ単位で適用**: トップ → 固定ページ → 記事の順に新デザインへ。各段階で本番反映可能に保つ。
4. **動的化は決済直前**: SSR / Functions 化は Stripe 導入（Phase 4）の直前にまとめて行い、それまで静的構成の利点（速さ・堅牢さ）を維持する。
5. **記録の保全**: 既存記事の frontmatter（`date` / `images` / `sourceUrl`）との後方互換を必ず保つ。

## 8. 開発

```bash
npm ci          # 依存インストール
npm run dev     # 開発サーバー (http://localhost:4321)
npm run build   # 静的ビルド (dist/)
npm run preview # ビルド成果物のローカル確認
```

- ブランチ運用・デプロイ・ディレクトリ構造・記事追加手順の詳細は [`CLAUDE.md`](./CLAUDE.md) を参照。
- デプロイは Cloudflare ダッシュボードの GitHub 連携（Workers Builds）。`main` への push / PR 作成で自動ビルド・公開。GitHub Actions (`ci.yml`) はビルド検証のみ。
- 本番 / プレビューの環境変数は、単一 Worker のビルド変数に `<NAME>_PRODUCTION` / `<NAME>_PREVIEW` を登録し、**ビルド時にブランチ（`WORKERS_CI_BRANCH`）で出し分ける**（純粋関数 `src/buildEnv.ts`）。詳細は [`CLAUDE.md` の「環境変数の本番 / プレビュー切り分け」](./CLAUDE.md) を参照。

### 環境変数

| 変数名                                             | 用途                                            | 種別             | 状態               |
| -------------------------------------------------- | ----------------------------------------------- | ---------------- | ------------------ |
| `PDFJS_EXPRESS_VIEWER`                             | PDF.js Express ビューワーライセンスキー         | 変数             | 現行               |
| `GA_MEASUREMENT_ID`                                | Google Analytics 測定 ID（gtag.js）             | 変数             | 現行               |
| `GOOGLE_MAPS_API_KEY`                              | Maps JavaScript API のブラウザキー（`/access`） | 変数             | 現行               |
| `PUBLIC_SITE_URL`                                  | canonical / OGP / 構造化データの基点 URL        | 変数             | 現行（既定値あり） |
| `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET` ほか | Stripe 決済                                     | シークレット     | Phase 4 で追加予定 |
| `CLERK_*`（公開鍵・シークレット鍵等）              | Clerk 認証                                      | 一部シークレット | Phase 5 で追加予定 |

> 本サイトは静的生成のため `PDFJS_EXPRESS_VIEWER` / `GA_MEASUREMENT_ID` / `GOOGLE_MAPS_API_KEY` は**ビルド時**に必要（ランタイム変数ではない）。デプロイ時は Cloudflare の **Settings > Build「Build variables and secrets」** に `<NAME>_PRODUCTION` / `<NAME>_PREVIEW` を登録し、ビルド時にブランチで出し分ける（プレビューに出したくない変数は `_PREVIEW` を未設定に）。`PUBLIC_SITE_URL` はプレビューが本番と別に索引されるのを防ぐため出し分けず、常に本番 URL を出す（未設定なら本番ドメインにフォールバック）。CI（GitHub Actions）は値に依存しないため未設定でよい（必要時のみ `vars` に素の名前で登録）。いずれも秘匿情報ではないため変数として扱い（`GOOGLE_MAPS_API_KEY` は HTTP リファラ制限で保護）、将来の Stripe/Clerk の秘密鍵はビルド変数に置かずランタイムのシークレットとする。
